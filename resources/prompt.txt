Prompt:
1. The "North Star" (Context & Goals)
Before the "how," everyone must understand the "why." This prevents "feature creep" where developers add things you didn't ask for.

Abstract: A 3-sentence summary of the feature.

User Stories: "As a user, I want to [action] so that [value]."

Non-Goals: Explicitly list what not to build. (e.g., "We are NOT supporting offline mode in this version.")

2. System Architecture & Flow
Visuals are the most effective way to ensure everyone sees the same "big picture."

Component Diagram: How the new feature interacts with the existing frontend, backend, and third-party APIs.

Sequence Diagram: A step-by-step flow of data between components for a specific action.

3. The Technical "Source of Truth"
This is the section that ensures "same results from everyone." You must define the interfaces.

A. Data Schema (The "What")
Don't just say "we need a table for users." Provide the exact field names and types:
| Field Name | Type | Constraints |
| :--- | :--- | :--- |
| user_id | UUID | Primary Key, Not Null |
| email | String(255) | Unique, Indexed |
| created_at | Timestamp | Default: current_utc |

B. API Contracts (The "How")
Define the exact endpoints. Use OpenAPI (Swagger) style:

Endpoint: POST /v1/orders

Payload: { "item_id": "string", "quantity": "int" }

Success Response: 201 Created

Error Cases: Define what happens on a 400 or 500 error.

4. Application "Bootstrap" Guide (The Scratch Setup)
Since you are setting this up from scratch, the design doc must include the Development Environment specs:

Tech Stack: Specific versions (e.g., Node.js v20.x, Python 3.11, PostgreSQL 15).

Folder Structure: Provide a tree diagram of how the code should be organized (e.g., /src/services, /src/models).

Boilerplate/Tooling: Specify the linting rules (ESLint/Prettier), testing frameworks (Jest/PyTest), and CI/CD triggers.

5. Implementation Requirements & Constraints
These are the "guardrails" that prevent developers from taking creative liberties that might hurt the system.

Security: "All passwords must be hashed using Argon2id."

Performance: "The API must respond within <200ms for 95% of requests."

Error Handling: "Log all 5xx errors to Sentry with the trace ID."

6. The "Definition of Done" (DoD)
To get the same result, you must define exactly what a "finished" task looks like:

Unit test coverage > 80%.

All API endpoints documented in Postman/Swagger.

Feature flag implemented for dark-launching.

Documentation updated in the README.
